# Product Requirements Document (PRD)

**โครงการ:** ระบบแจ้งปัญหา IT ภายใน (Internal IT Helpdesk System)
**เวอร์ชัน:** 1.0
**วันที่:** 17 พฤศจิกายน 2025
**ผู้จัดทำ:** Gemini (Project Manager)
**Stakeholders:** แผนก IT, หัวหน้าแผนกต่างๆ, ผู้บริหาร

---

## 1. บทนำ (Introduction)

เอกสารนี้สรุปข้อกำหนดสำหรับ "ระบบแจ้งปัญหา IT ภายใน" ซึ่งเป็นระบบ Web-based ที่มีจุดประสงค์เพื่อเป็นช่องทางหลัก (Single Source of Truth) ในการรับแจ้ง, ติดตาม และแก้ไขปัญหาด้าน IT ทั้งหมดภายในองค์กร แทนที่วิธีการเดิมที่กระจัดกระจาย (เช่น การแจ้งผ่านโทรศัพท์, อีเมล หรือ Chat)

## 2. ปัญหาที่ต้องการแก้ไข (Problem Statement)

ปัจจุบัน บริษัทไม่มีระบบกลางในการจัดการคำร้องขอความช่วยเหลือด้าน IT ส่งผลให้เกิดปัญหา:
* **ฝั่งผู้ใช้งาน (User):** ไม่ทราบว่าปัญหาที่แจ้งไปถึงไหน ใครกำลังดำเนินการ ต้องเสียเวลาติดตามงานซ้ำซ้อน
* **ฝั่ง IT (Admin):** จัดการงานลำบาก ไม่มีระบบ Prioritize งานที่ชัดเจน (งานด่วน/งานแทรก), ทำงานซ้ำซ้อน, และไม่มีข้อมูลสถิติว่าปัญหาใดเกิดขึ้นบ่อย
* **ฝั่งองค์กร:** สิ้นเปลืองเวลาทำงานของพนักงาน (Productivity Loss) และไม่สามารถวัดผลประสิทธิภาพของทีม IT ได้

## 3. เป้าหมายและวัตถุประสงค์ (Goals & Objectives)

1.  **เพิ่มประสิทธิภาพ (Efficiency):** ลดเวลาเฉลี่ยในการแก้ไขปัญหา (Average Resolution Time) ลง 30% ภายใน 6 เดือน
2.  **สร้างความโปร่งใส (Transparency):** ผู้ใช้งานสามารถติดตามสถานะการแก้ไขปัญหาของตนเองได้ 100%
3.  **รวบรวมข้อมูล (Data-Driven):** สร้างฐานข้อมูลปัญหา (Knowledge Base) และรายงานสถิติ เพื่อใช้ในการวิเคราะห์และวางแผนป้องกันปัญหาในอนาคต
4.  **เพิ่มความพึงพอใจ (Satisfaction):** เพิ่มคะแนนความพึงพอใจของผู้ใช้งานต่อการบริการของแผนก IT

## 5. คุณสมบัติหลักและ User Stories (Features & User Stories)

#### ส่วนที่ 1: ผู้ใช้งานทั่วไป (User Features)

**Epic 1.0: การแจ้งปัญหา (Ticket Submission)**
* **User Story 1.1:** *ในฐานะผู้ใช้งาน*, ฉันต้องการ *กรอกแบบฟอร์มแจ้งปัญหาที่เรียบง่าย* (เลือกประเภท, ระบุอาการ, แนบรูป) เพื่อให้ IT ได้รับข้อมูลครบถ้วนในครั้งเดียว
* **User Story 1.2:** *ในฐานะผู้ใช้งาน*, ฉันต้องการ *ได้รับอีเมลยืนยันอัตโนมัติ* พร้อมเลขที่ Ticket เมื่อแจ้งปัญหาสำเร็จ เพื่อใช้อ้างอิง

**Epic 2.0: การติดตามสถานะ (Ticket Tracking)**
* **User Story 2.1:** *ในฐานะผู้ใช้งาน*, ฉันต้องการ *เห็นรายการ Ticket ทั้งหมดที่ฉันเคยแจ้ง* พร้อมสถานะปัจจุบัน (เช่น: ใหม่, กำลังแก้, แก้ไขแล้ว) เพื่อจะได้ไม่ต้องโทรถาม
* **User Story 2.2:** *ในฐานะผู้ใช้งาน*, ฉันต้องการ *สื่อสารโต้ตอบกับ Admin* ในหน้ารายละเอียด Ticket นั้นๆ (เช่น ตอบคำถามเพิ่ม)
* **User Story 2.3:** *ในฐานะผู้ใช้งาน*, ฉันต้องการ *กดยืนยันปิดงาน (Close)* หรือ *แจ้งเปิดใหม่ (Re-open)* หากปัญหายังไม่จบ

**Epic 3.0: คลังความรู้ (Knowledge Base)**
* **User Story 3.1:** *ในฐานะผู้ใช้งาน*, ฉันต้องการ *ค้นหาปัญหาที่พบบ่อย (FAQ)* (เช่น "วิธีแก้ปัญหา Wi-Fi") ก่อนแจ้ง Ticket เพื่อที่ฉันจะสามารถแก้ปัญหาเองได้

#### ส่วนที่ 2: ผู้ดูแลระบบ (Admin Features)

**Epic 4.0: แดชบอร์ด (Admin Dashboard)**
* **User Story 4.1:** *ในฐานะ Admin*, ฉันต้องการ *เห็นแดชบอร์ดสรุป* (เช่น: Ticket ใหม่, Ticket ค้าง, Ticket ทั้งหมด) ทันทีที่ล็อกอิน เพื่อประเมินสถานการณ์

**Epic 5.0: การจัดการ Ticket (Ticket Queue Management)**
* **User Story 5.1:** *ในฐานะ Admin*, ฉันต้องการ *ดู Ticket ทั้งหมดในคิว* และสามารถ *กรอง (Filter)* ตามสถานะ, Priority, หรือผู้แจ้งได้
* **User Story 5.2:** *ในฐานะ Admin*, ฉันต้องการ *รับงาน (Assign) Ticket* ให้ตัวเอง หรือส่งต่อให้ IT คนอื่น
* **User Story 5.3:** *ในฐานะ Admin*, ฉันต้องการ *เปลี่ยนสถานะ* (เช่น: In Progress, Pending, Resolved) และ *เปลี่ยน Priority* ของ Ticket ได้
* **User Story 5.4:** *ในฐานะ Admin*, ฉันต้องการ *บันทึกข้อความภายใน (Internal Note)* ที่ User มองไม่เห็น เพื่อใช้สื่อสารกันเองในทีม IT

**Epic 6.0: การจัดการคลังความรู้ (KB Management)**
* **User Story 6.1:** *ในฐานะ Admin*, ฉันต้องการ *สร้าง, แก้ไข, หรือลบ* บทความใน Knowledge Base เพื่อให้ข้อมูลทันสมัย

**Epic 7.0: รายงาน (Reporting)**
* **User Story 7.1:** *ในฐานะ IT Manager*, ฉันต้องการ *ดูรายงานสรุป* (เช่น: จำนวน Ticket รายเดือน, ปัญหาที่พบบ่อยสุด, เวลาเฉลี่ยในการแก้ไข) เพื่อวัดผลการทำงาน

## 6. ข้อกำหนดด้านฟังก์ชัน (Functional Requirements)

* **FR-1: การยืนยันตัวตน (Authentication)**
    * ระบบต้องเชื่อมต่อกับ Active Directory (AD) ของบริษัทเพื่อใช้ Single Sign-On (SSO)
    * ระบบต้องแยกระดับสิทธิ์ (Role) ระหว่าง "User" และ "Admin" ชัดเจน

* **FR-2: แบบฟอร์มแจ้งปัญหาใหม่ (New Ticket Form)**
    * ต้องมีช่องบังคับ (Required Fields): ประเภทปัญหา (Category), หัวข้อ (Subject), รายละเอียด (Description)
    * ต้องรองรับการแนบไฟล์ (เช่น .jpg, .png, .zip) ขนาดไม่เกิน 10MB
    * เมื่อกด Submit ระบบต้องสร้าง Ticket ID ที่ไม่ซ้ำกัน (เช่น IT-2025-0001)

* **FR-3: ระบบแจ้งเตือน (Notification)**
    * ต้องส่งอีเมลแจ้งเตือนอัตโนมัติ (ไปยัง User และ Admin) ในเหตุการณ์ต่อไปนี้:
        * (User) เมื่อสร้าง Ticket ใหม่
        * (User) เมื่อ Admin ตอบกลับ หรือ แก้ไขเสร็จ (Resolved)
        * (Admin) เมื่อมี Ticket ใหม่เข้ามา หรือ User ตอบกลับ

* **FR-4: สถานะของ Ticket (Ticket Statuses)**
    * ระบบต้องมีสถานะอย่างน้อยดังนี้:
        1.  **Open:** (ใหม่) ยังไม่มีคนรับ
        2.  **In Progress:** (กำลังดำเนินการ) Admin รับงานแล้ว
        3.  **Pending:** (รอข้อมูล) Admin รอข้อมูล/อะไหล่/การตอบกลับจาก User
        4.  **Resolved:** (แก้ไขแล้ว) Admin แก้ไขเสร็จ รอ User ยืนยัน
        5.  **Closed:** (ปิดงาน) User ยืนยัน หรือระบบปิดอัตโนมัติหลัง Resolved 3 วัน
        6.  **Canceled:** (ยกเลิก)

## 7. ข้อกำหนดที่ไม่ใช่ฟังก์ชัน (Non-Functional Requirements)

* **ความปลอดภัย (Security):** ข้อมูลทั้งหมดต้องเข้ารหัส (HTTPS) และป้องกันการเข้าถึงข้อมูล Ticket ของผู้อื่น (User เห็นได้เฉพาะ Ticket ตัวเอง)
* **ประสิทธิภาพ (Performance):** หน้าเว็บต้องโหลดเสร็จสมบูรณ์ภายใน 3 วินาที
* **การใช้งาน (Usability):** User ต้องสามารถสร้าง Ticket ใหม่ได้ภายใน 3 คลิกจากหน้าแรก
* **ความเข้ากันได้ (Compatibility):** ต้องรองรับ Web Browser มาตรฐาน (Chrome, Firefox, Edge) เวอร์ชันล่าสุด

## 8. ข้อจำกัดและข้อสมมติฐาน (Constraints & Assumptions)

* **ข้อจำกัด:** ระบบนี้ต้องพัฒนาเป็น Web Application เท่านั้น (ยังไม่มี Mobile App ในเฟสนี้)
* **ข้อจำกัด:** ต้องใช้ระบบ Email Server ภายในของบริษัทในการส่งแจ้งเตือน
* **ข้อสมมติฐาน:** พนักงานทุกคนมี Account AD สำหรับการ Login

## 9. ตัวชี้วัดความสำเร็จ (Success Metrics)

* **Metric 1:** อัตราการรับงาน (Ticket Adoption Rate) > 90% (วัดจากการแจ้งปัญหาผ่านระบบ เทียบกับการแจ้งผ่านช่องทางเดิม)
* **Metric 2:** เวลาเฉลี่ยในการแก้ไขปัญหา (Average Resolution Time) ลดลงตามเป้าหมาย
* **Metric 3:** คะแนนความพึงพอใจของ User (CSAT) > 4.5/5 (วัดจากแบบสอบถามหลังปิด Ticket)

## 10. สิ่งที่ไม่อยู่ในขอบเขต (Out of Scope - for V2)

* ระบบจัดการทรัพย์สิน IT (IT Asset Management)
* Chatbot ตอบปัญหาอัตโนมัติ
* Mobile Application
* การตั้งค่า SLA (Service Level Agreement) แบบละเอียด
